# Goal 8: Metadata validation gate (rules + stats + human review before KB publish)

## 1. Outcome
Implement a metadata validation gate so autogenerated captions/labels (especially for multimodal KB items) are verified, auditable, and safe before they affect resident-facing retrieval.

**What changes:**  
KB publish is gated by a validation pipeline that classifies metadata as **approved vs quarantined**, with optional offline/batch LLM support for review (not in the hot path).

**Who benefits:**
- Residents: safer, more reliable filtering  
- Admins: clear review queue  
- Maintainers: reproducible KB versions with auditable metadata quality  

**What success looks like:**
- Low-confidence / rule-failing metadata is blocked from publish (or excluded from resident retrieval) until reviewed  
- Review decisions are recorded and traceable to a KB version  
- Validation outputs (rules + stats) are inspectable and consistent for the same KB snapshot/config  

---

## 2. Why this matters

**Current limitation:**  
No formal publish gate for autogenerated metadata/captions.

**Risk addressed:**  
Bad metadata breaks Goal 7 filtering/scoping and increases risk for medical/safety content.

**Value:**  
Transforms “best-effort metadata” into trusted, versioned retrieval constraints with auditable human oversight.

---

## 3. Scope

### Validation Rules & Checks
- Captions: text quality + safety checks  
- Labels/taxonomy assignments (Goal 7) with confidence/coverage  

### Workflow
- Failed items → **quarantined for review**
- Only approved metadata affects resident-facing retrieval  

### Admin Review (MVP)
- View failures with evidence (rules, stats)  
- Approve / reject / override with reason codes  
- Track audit history tied to KB version  

### Optional LLM Judge (Offline Only)
- Assists human reviewers  
- Never blocks hot path  

---

## 4. Non-goals
- No LLM judge in resident request path  
- No fully automated publishing for high-risk metadata  
- No full annotation platform  
- No KB restructuring or re-chunking  

---

## 5. System Impact

### a. Data / Schema

**Validation Status (per item):**
- `approved`
- `quarantined`
- `needs_review`
- `rejected`

**Persisted Validation Results:**
- Rules fired (IDs, severity, messages)  
- Statistical checks (values, thresholds, pass/fail)  
- Model provenance (captioner/labeler version)  

**Review Outcomes:**
- Reviewer identity  
- Timestamp  
- Decision + reason code  
- Notes  
- KB version link  

**Goal 7 Relationship:**
- Applies to taxonomy and filtering metadata  

---

### b. API / Interfaces

**Publish Flow:**
- Returns `pass | blocked` with summary  

**Admin Review:**
- List quarantined items  
- Fetch validation details  
- Apply decisions  

**Retrieval:**
- Quarantined metadata is ignored for enforcement  

---

### c. UX / Behavior

**Admin:**
- “Ready to publish” vs “Blocked by validation”  
- Review queue with filters  

**Resident:**
- No UI change; improved safety via backend filtering  

---

## 6. Integration Points

**Depends on:**
- Goal 7 (metadata schema + filtering policy)  
- KB versioning  

**Affects:**
- Goal 4/5 (retrieval behavior)  
- Goal 10 (metrics/logging)  
- Goal 11 (cache invalidation)  

---

## 7. Edge Cases / Failure Modes

- **Validation down:** fail closed or require auditable override  
- **High quarantine volume:** batching + severity sorting  
- **Rule conflicts:** allow overrides with logging  
- **Taxonomy drift:** quarantine unknown IDs  
- **Multimodal mismatch:** quarantine with mismatch code  

---

## 8. Logging & Metrics

### Validation Run
- KB version, timestamp  
- Rules + stats results  
- Counts: total, quarantined, approved, rejected  
- Breakdown by modality + taxonomy  

### Review Workflow
- Decision logs with reason codes  
- Time-to-approve  
- Backlog size  

### Publish
- Blocked vs passed  
- Quarantine summary  

---

## 9. Determinism / Constraints

- Validation must be deterministic for same inputs  
- Retrieval must not use quarantined metadata  
- Overrides must be explicit and auditable  

---

## 10. Definition of Done (DoD)

- Validation rules implemented (captions + labels)  
- Per-item validation status persisted  
- Publish is gated  
- Admin review workflow exists  
- Optional LLM judge available (non-blocking)  
- Metrics/logging implemented  

---

## 11. Open Decisions

- Storage format (tables vs JSON)  
- Gate behavior:
  - Block publish (default)  
  - Or allow publish with quarantined metadata inactive  
- Rule thresholds (especially for safety content)

# Policy / Rules

## Gate Policy (Authoritative)

**Default:**  
Publish is blocked if quarantined items exist.

**Alternative (auditable only):**
- Publish allowed  
- Quarantined metadata inactive in retrieval  
- Console shows warning  

---

## Rule Categories

### Taxonomy Integrity
- Valid taxonomy IDs required  
- Primary assignment required  

### Authority/Scope Integrity
- Must match allowed enums  
- Missing required values → quarantine  

### Caption Sanity
- Non-empty  
- English  
- No placeholders/irrelevant boilerplate  

### Safety-Critical Checks
- Medical/safety items require stricter thresholds  
- Mandatory human approval on first publish  

---

## Workflow / Review States

### States
- **approved** → usable in retrieval  
- **quarantined** → blocked  
- **rejected** → must be corrected  

### Transitions
- quarantined → approved  
- quarantined → rejected  
- rejected → quarantined (after re-validation)  

All transitions must include:
- Reviewer identity  
- Reason code  
- Audit log  

---
